### Medium

**[M-01] A09 — No WARNING log for invalid/missing endpoint (404 path)**
`Classes/Middleware/ApiResolverMiddleware.php:79–80`

When `getEndpoint()` returns `null`, the middleware silently forwards the request with `$handler->handle($request)` without emitting any log entry. The security guidelines require a `WARNING` be logged for "endpoint not found" before returning 404. Because no 404 is even explicitly returned here (the request is forwarded to the next handler), the security-relevant event is invisible in the log channel.

Remediation: Before calling `$handler->handle($request)` on the "no endpoint found" branch, log a `WARNING` to the `simple_rest_api` channel with the requested HTTP method and path.

---

**[M-02] A09 — No WARNING log before returning 400 on coercion failure**
`Classes/Collection/EndpointParameterResolver.php:50–58`

The `buildMethodParameters()` method performs type coercion with a `match` expression, but if `$param->type` falls through to the `default` branch (unknown type), no exception is thrown and no log entry is produced. The security guidelines require a `WARNING` log entry when coercion fails. Additionally, for types other than `int`, `string`, `float`, `bool`, and `ServerRequestInterface`, there is no validation that rejects the value or logs the anomaly. Per the guidelines, non-scalar parameter types should be rejected with a 400 and a `WARNING`.

Remediation: Add explicit handling for the `default` branch — throw an `InvalidParameterException` (returning 400) and emit a `WARNING` log entry to the `simple_rest_api` channel before doing so.

---

**[M-03] A03 — Type-coercion silently succeeds for semantically invalid values**
`Classes/Collection/EndpointParameterResolver.php:50–58`

The `int` coercion is performed with a bare `(int)$value` cast. In PHP, `(int)"abc"` yields `0` rather than an error. The security guidelines state that `InvalidParameterException` (400) must be thrown when coercion fails. The current implementation allows values like `"abc"` cast to `int` to silently become `0`, passing an unintended value to consumer methods without any error response.

Remediation: Use `filter_var($value, FILTER_VALIDATE_INT)` (and equivalent validators for `float` and `bool`) rather than bare PHP casts. If validation fails, throw `InvalidParameterException` and emit a `WARNING` log entry.

### Low / Informational

**[L-01] A01 — No documentation warning on `AsApiEndpoint` attribute**
`Classes/Attribute/AsApiEndpoint.php`

The `AsApiEndpoint` attribute has no inline documentation warning that methods annotated with it are accessible without authentication by default. Consumers may incorrectly assume the attribute itself enforces access control. The security guidelines require that consumer responsibilities are documented within the attribute.

Remediation: Add a PHPDoc block to `AsApiEndpoint` explicitly stating that no authentication or authorization is provided by the extension and that the consumer must implement these in a middleware or within the endpoint method.

---

**[L-02] A05 — `CacheHashFixer` mutates global config on path-prefix match only**
`Classes/Middleware/CacheHashFixer.php:39–42`

`CacheHashFixer` disables `pageNotFoundOnCHashError` and `cacheHash.enforceValidation` globally (via `$GLOBALS['TYPO3_CONF_VARS']`) whenever the request path starts with the API base path. This is correct by design, but the mutation is permanent for the request lifecycle. There is no restoration of the original values. If TYPO3 ever re-reads these settings mid-request for non-API sub-requests (e.g. sub-requests initiated by TYPO3 internals), the relaxed cache-hash settings will apply unexpectedly.

Remediation: Store the original values before overwriting and restore them after the handler returns, or confirm through TYPO3 core knowledge that these globals are only read once per request at the point where they matter.

---

**[L-03] A04 — `error_log()` used in example event listener**
`Classes/EventListener/ApiResponseModifierExample.php:99`

The example event listener uses `error_log()` instead of TYPO3's logging API. While this file is example/dev code, it sets a poor pattern for consumers and could appear in production environments if the class is inadvertently registered as an event listener.

Remediation: Replace `error_log()` with TYPO3's `LogManager`/`LoggerInterface` in the example, or add a clear `@internal` / `@example` annotation and exclude the file from production DI configuration.

---

**[L-04] A06 — Abandoned packages in dependency tree**
`composer.json` (transitive dependencies)

`composer audit` reports two abandoned packages as informational notices:
- `azjezz/psl` — suggested replacement: `php-standard-library/php-standard-library`
- `doctrine/annotations` — no replacement suggested

These are transitive dependencies (not direct `require` entries). No known CVEs are associated. Risk is low but increases over time as abandoned packages receive no security patches.

Remediation: Monitor whether upstream dependencies that pull in these packages provide alternatives. No immediate action required.

## OWASP Checklist

### A01 — Broken Access Control
- [x] No endpoint is accessible without the consumer implementing authentication
- [ ] Consumer security responsibilities are documented in the extension docs (gap: missing from `AsApiEndpoint` attribute inline docs — see L-01)
- [x] No TYPO3 page access restrictions are inadvertently bypassed

**Notes:** The extension correctly has no built-in access control by design. `ApiResolverMiddleware` runs after `prepare-tsfe-rendering`, meaning TYPO3's frontend bootstrap is in place. There is no mechanism to bypass TYPO3 page-level access restrictions from within the extension itself. The middleware order in `Configuration/RequestMiddlewares.php` is correct. The gap is documentation-only (L-01).

### A03 — Injection
- [ ] All URL parameters are type-coerced before being passed to consumer methods (gap: bare PHP casts allow silent coercion of invalid values — see M-03)
- [x] No raw URL parameter values are used in TYPO3 queries or system calls within the extension
- [ ] `InvalidParameterException` (400) is thrown on coercion failure (gap: bare casts do not throw — see M-03)

**Notes:** The extension does not use any database queries or shell calls internally. All parameter routing is purely string-based path splitting. The injection risk is limited to values passed through to consumer code, but the silent cast behaviour (M-03) means consumers may receive `0` instead of a type error when non-numeric strings are passed for `int` parameters. The `default` branch of the `match` in `EndpointParameterResolver` passes through the raw string value for unknown types (M-02).

### A04 — Insecure Design
- [x] Extension does not handle auth, authz, or rate limiting (by design — documented in CLAUDE.md and security.md)
- [ ] Consumer responsibility documentation is present and up to date (gap: not present in attribute PHPDoc — see L-01)
- [x] No new features implicitly assume security guarantees not provided by the extension

**Notes:** The `ModifyApiResponseEvent` comment block helpfully lists CORS header addition as a common use case, which is a consumer responsibility and appropriate for documentation. The `BeforeParameterMappingEvent` and `AfterParameterMappingEvent` allow consumers to inject arbitrary parameters or replace the full parameter array — this is intentional design, and the consumer is responsible for validating anything injected via these events.

### A05 — Security Misconfiguration
- [x] Base path validation enforces correct format (`/name/`)
- [x] Route enhancer configuration is validated
- [ ] Default configuration exposes minimum necessary surface (note: see L-02 regarding global config mutation)

**Notes:** `ExtensionConfiguration.isValidBasePath()` uses the regex `^\/([a-zA-Z0-9_-]+\/)+$` which correctly enforces start-slash, end-slash, and safe characters. Invalid configurations silently fall back to `/api/` — this is safe behaviour. The route enhancer correctly uses `{params?}` with `.*` requirement to catch all sub-paths; this is intentional and necessary for operation. The middleware order places `ApiAspectMiddleware` before `CacheHashFixer` and `ApiResolverMiddleware`, which is correct. The `CacheHashFixer` mutates global config permanently (L-02), which is a low-severity concern.

### A06 — Vulnerable Components
- [x] `composer audit` output: no known vulnerabilities
- [x] All dependencies are justified and minimized
- [x] No new unjustified dependencies added since last audit

**Notes:** All direct `require` entries are standard PSR interfaces, Symfony DI, Symfony polyfill (required for PHP 8.0 compat), and TYPO3 core packages. No unexpected or unjustified dependencies found. The two abandoned transitive packages (L-04) carry no CVEs at audit date.

### A09 — Security Logging
- [ ] Invalid parameter type triggers a WARNING log entry (gap: M-02, M-03)
- [ ] Endpoint not found triggers a WARNING log entry (gap: M-01)
- [ ] Log channel `simple_rest_api` is used consistently (gap: no logging exists at all in the extension for security-relevant events)

**Notes:** There is **no logging at all** in the `Classes/` directory for security-relevant events. Neither `ApiResolverMiddleware`, `EndpointParameterResolver`, nor any other class imports or uses TYPO3's `LoggerInterface` or `LogManager`. This means all three A09 requirements from the security guidelines are unmet. This is the most significant systematic gap found in this audit.

## Recommendations

1. **[Priority: High] Implement security-relevant logging (A09).** None of the three required logging points are implemented. Add `LoggerInterface` injection to `ApiResolverMiddleware` and `EndpointParameterResolver`, emit `WARNING` logs for endpoint-not-found and parameter coercion failures, and use the `simple_rest_api` log channel. This is required by the security guidelines.

2. **[Priority: High] Replace bare PHP casts with validated coercion (A03/M-03).** Use `filter_var()` with appropriate `FILTER_VALIDATE_*` flags for `int`, `float`, and `bool` parameters. Throw `InvalidParameterException` (400) when validation fails rather than silently coercing invalid values.

3. **[Priority: Medium] Add security notice to `AsApiEndpoint` attribute (A01/A04/L-01).** A short PHPDoc warning on the attribute is the natural place for consumers to learn about the extension's security boundary. This is low effort and high value for consumer security posture.

4. **[Priority: Low] Review `CacheHashFixer` global mutation (A05/L-02).** Confirm with TYPO3 internals knowledge whether `$GLOBALS['TYPO3_CONF_VARS']` values are read only once per request at the relevant point, or add a try/finally block to restore original values.

5. **[Priority: Low] Clean up `ApiResponseModifierExample.php` (A04/L-03).** Either replace `error_log()` with TYPO3's logging API, add explicit `@internal` markers, or ensure the class is excluded from production DI configuration.
