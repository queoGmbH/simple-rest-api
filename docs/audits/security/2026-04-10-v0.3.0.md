## Resolved Findings (from v0.2.4)

All 7 findings from the v0.2.4 audit have been resolved in this release cycle via dedicated
security branches.

### M-01 — A09: No WARNING log for invalid/missing endpoint (404 path)

**Branch:** `security/1-security-logging`
**Resolution:** `ApiResolverMiddleware` now accepts a `LoggerInterface` via constructor injection
(autowired by TYPO3's `LoggerInterfacePass`). A `WARNING` is emitted to the `simple_rest_api`
channel when `getEndpoint()` returns `null` — i.e. the API base path was matched but no
registered endpoint could be found for the requested method/path combination.

### M-02 — A09: No WARNING log before returning 400 on coercion failure

**Branch:** `security/1-security-logging`
**Resolution:** `ApiResolverMiddleware` catches `InvalidParameterException` thrown by
`EndpointParameterResolver`, logs a `WARNING` with the parameter name and raw value, and
returns a `400 Bad Request` JSON response.

### M-03 — A03: Type-coercion silently succeeds for semantically invalid values

**Branch:** `security/2-validated-type-coercion`
**Resolution:** Bare PHP casts (`(int)`, `(float)`, `(bool)`) replaced with validated coercion
in `EndpointParameterResolver::coerce()`. Uses `filter_var()` with `FILTER_VALIDATE_INT`,
`FILTER_VALIDATE_FLOAT`, and `FILTER_VALIDATE_BOOLEAN` (with `FILTER_NULL_ON_FAILURE`).
Empty strings are explicitly rejected for all numeric and boolean types.
`InvalidParameterException` is thrown on validation failure with a descriptive message
containing the parameter name. Test coverage added via `#[DataProvider]` with 16 valid and
8 invalid coercion cases.

### L-01 — A01/A04: No documentation warning on `AsApiEndpoint` attribute

**Branch:** `security/3-asapiendpoint-security-notice`
**Resolution:** `AsApiEndpoint` now carries a PHPDoc block with an `@api` tag and an explicit
`SECURITY NOTICE` section documenting that authentication, authorization, rate limiting, and
input validation beyond scalar type coercion are the consumer's responsibility.

### L-02 — A05: `CacheHashFixer` mutates global config without restoring

**Branch:** `security/4-cachehashfixer-restore-conf-vars`
**Resolution:** `CacheHashFixer` now stores original values of `pageNotFoundOnCHashError` and
`cacheHash.enforceValidation` before overwriting them and restores both in a `finally` block,
guaranteeing restoration even if the downstream handler throws.

### L-03 — A04: `error_log()` used in example event listener

**Branch:** `security/5-replace-error-log-in-example`
**Resolution:** The `logResponse()` method (which contained the `error_log()` call) was
removed from `ApiResponseModifierExample`. The remaining methods already demonstrate data
access patterns adequately, so removal was cleaner than replacement.

### L-04 — A06: Abandoned packages in dependency tree

**Status:** Carry-over (informational, no action required)
**Notes:** `azjezz/psl` and `doctrine/annotations` remain as transitive dev dependencies pulled
in by TYPO3 core packages. No CVEs associated. This finding remains open until upstream TYPO3
provides alternatives.

## OWASP Checklist

### A01 — Broken Access Control
- [x] No endpoint is accessible without the consumer implementing authentication
- [x] Consumer security responsibilities documented in `AsApiEndpoint` attribute PHPDoc (resolved L-01)
- [x] No TYPO3 page access restrictions are inadvertently bypassed

### A03 — Injection
- [x] All URL parameters validated with `filter_var()` before passing to consumer methods (resolved M-03)
- [x] `InvalidParameterException` (400) thrown on coercion failure (resolved M-03)
- [x] No raw URL parameter values used in TYPO3 queries or system calls

### A04 — Insecure Design
- [x] Extension does not handle auth, authz, or rate limiting (by design — documented)
- [x] Consumer responsibility documentation present in attribute PHPDoc (resolved L-01)
- [x] Example code uses appropriate logging patterns (resolved L-03)

### A05 — Security Misconfiguration
- [x] Base path validation enforces correct format
- [x] Route enhancer configuration validated
- [x] `CacheHashFixer` restores original config values in `finally` (resolved L-02)

### A06 — Vulnerable Components
- [x] `composer audit` output: no known vulnerabilities
- [x] All dependencies justified and minimized
- [x] `psr/log: ^3.0` added as explicit dependency (was implicit via TYPO3 core)

### A09 — Security Logging
- [x] Invalid parameter type triggers WARNING log entry (resolved M-02, M-03)
- [x] Endpoint not found triggers WARNING log entry (resolved M-01)
- [x] Log channel `simple_rest_api` used consistently
