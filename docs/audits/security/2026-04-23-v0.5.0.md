## Changes Since v0.4.0

Two runtime files changed since the v0.4.0 tag. All other changes are tests,
documentation, and CI configuration with no security impact.

### `Classes/Middleware/ApiResolverMiddleware.php`

Two changes:

1. **ExtensionConfiguration instantiation moved into `process()`.**
   Previously `ExtensionConfiguration` was injected via the constructor (typed to
   the concrete class). It is now instantiated per-request via
   `GeneralUtility::makeInstance(ExtensionConfiguration::class, $request)`, which
   passes the current PSR-7 request so site-specific base-path settings are resolved
   correctly. This is a correctness fix, not a security change. The configuration
   object is still validated before use (`isValidBasePath()`) and no user-controlled
   value reaches the middleware without going through that validation.

2. **Endpoint-not-found path returns `JsonResponse(['error' => 'Not Found'], 404)`
   instead of forwarding to `$handler->handle($request)`.**
   The previous behaviour forwarded the request down the middleware stack, which
   could in theory allow an API-prefixed URL to reach unexpected handlers if the
   route enhancer matched but no endpoint was registered. The new behaviour
   terminates the response immediately. This is a **security improvement**: it
   eliminates any possibility of fall-through to unintended page-rendering logic
   for unrecognised API paths. The WARNING log entry is still emitted before the
   404 response, which satisfies the A09 logging requirement.

### `Configuration/Services.php`

Added a `Development`-context guard that loads `Dev/` classes
(`E2eTestController`, `E2eTestResponseListener`) only when
`Environment::getContext()->isDevelopment()` is true. These classes were already
present in the repository but previously always loaded.

Security assessment: The guard uses TYPO3's `Environment::getContext()` API, which
reads the context from the `TYPO3_CONTEXT` environment variable set at server level.
This is the standard TYPO3 mechanism and cannot be spoofed via HTTP. The `Dev/`
classes therefore cannot be loaded in a production (`Production/`) or staging context.
This change **improves** the security posture by ensuring test endpoints and event
listeners are never active outside a development environment.

The `E2eTestController` endpoints (`/e2e/ping`, `/e2e/params/*`) expose no sensitive
data and perform no privileged operations; their risk surface when accessible is
limited to confirming that routing works — but the context guard makes them
unreachable in production regardless.

### Security-relevant finding: none

No new attack vectors introduced. Two of the three changes improve the security
posture (hard 404 for unknown API paths; context-guarded Dev endpoints).

## OWASP Checklist

### A01 — Broken Access Control
- [x] No endpoint is accessible without the consumer implementing authentication
- [x] Consumer security responsibilities are documented in the extension docs
- [x] No TYPO3 page access restrictions are inadvertently bypassed

**Notes:** Unchanged since v0.4.0. The new hard-404 behaviour for unknown API paths
reduces the surface for accidental fall-through to TYPO3 page rendering. The
`E2eTestController` endpoints are correctly gated behind the Development context
check and cannot be reached in production.

### A03 — Injection
- [x] All URL parameters are type-coerced before being passed to consumer methods
- [x] No raw URL parameter values are used in TYPO3 queries or system calls within the extension
- [x] `InvalidParameterException` (400) is thrown on coercion failure

**Notes:** No changes to parameter handling in this release. The coercion logic in
`EndpointParameterResolver` (int, float, bool via `filter_var`, string pass-through)
is unchanged and remains correct. The `default` branch in `coerce()` passes unknown
types through as raw strings — this is an accepted design trade-off since PHPStan
level 9 and strict types prevent the extension itself from using those values
unsafely, and the responsibility for unknown type handling is documented for consumers.

### A04 — Insecure Design
- [x] Extension does not handle auth, authz, or rate limiting (by design — documented)
- [x] Consumer responsibility documentation is present and up to date
- [x] No new features implicitly assume security guarantees not provided by the extension

**Notes:** The SECURITY NOTICE in `AsApiEndpoint` attribute docblock remains in place.
No new features introduce implicit security assumptions.

### A05 — Security Misconfiguration
- [x] Base path validation enforces correct format (`/name/`)
- [x] Route enhancer configuration is validated
- [x] Default configuration exposes minimum necessary surface

**Notes:** `ExtensionConfiguration::isValidBasePath()` uses pattern
`/^\/([a-zA-Z0-9_-]+\/)+$/` which requires the path to start and end with `/` and
restricts segment characters to letters, digits, hyphens, and underscores. Invalid
values silently fall back to the default `/api/` — this is correct behaviour (fail
closed). No configuration changes in this release.

### A06 — Vulnerable Components
- [x] `composer audit` output: no known vulnerabilities
- [x] All dependencies are justified and minimized
- [x] No new unjustified dependencies added since last audit

**Notes:** `composer audit` reports no security advisories. Two abandoned transitive
packages remain (carry-over from v0.4.0, see Dependency Scan below). No new
`require` entries were added in this release.

### A09 — Security Logging
- [x] Invalid parameter type triggers a WARNING log entry
- [x] Endpoint not found triggers a WARNING log entry
- [x] Log channel `simple_rest_api` is used consistently

**Notes:** Both WARNING log paths are present and verified in
`ApiResolverMiddleware::process()`:
- Lines 60–68: `$this->logger->warning(...)` on `InvalidParameterException` before
  returning 400.
- Lines 96–103: `$this->logger->warning(...)` on endpoint not found before returning
  404.
The logger is injected as `LoggerInterface` via TYPO3's DI, which automatically
channels it to `simple_rest_api` based on the middleware class namespace. No
changes to logging in this release.
